# 백엔드-iOS 연동 작업 흐름 — 2026년 4월 22일 기준

## 목적

- 백엔드와 iOS가 같은 기능 단위를 같은 순서로 구현하고 검증할 수 있도록 공통 작업 흐름을 고정한다.
- "백엔드는 끝났는데 iOS가 못 붙는 상태", "iOS 화면은 있는데 서버 계약이 미정인 상태"를 줄인다.
- 다음 작업부터는 이 문서를 기준으로 각 플랫폼 TODO와 현재 상태 문서를 함께 업데이트한다.

## 현재 연동 판단

- 인증, 사용자, 운동 기록, 식단 기록, 신체 측정, 진행 사진, 목표 진행률까지 핵심 MVP 흐름은 이미 연동 가능한 수준이다.
- iOS `APIEndpoint`와 현재 백엔드 컨트롤러 경로는 큰 충돌 없이 맞아 있다.
- 지금 필요한 것은 새로운 대규모 계약 변경보다, 남은 기능을 같은 단위로 닫아가는 작업 운영 방식 정리다.

## 현재 기준 Source of Truth

### 계약 기준

- API 경로 및 요청/응답 구현 기준: 백엔드 컨트롤러 + DTO + iOS `APIEndpoint`/모델
- 장기 API 설명 문서: `docs/design-docs/API_DESIGN.md`
- 실제 진행 우선순위: `docs/CURRENT_STATUS.md`
- 백엔드 실행 TODO: `docs/exec-plans/BACKEND_TODO.md`
- iOS 실행 TODO: `docs/exec-plans/IOS_TODO.md`

### 작업 운영 원칙

- 계약의 최종 truth는 "문서"만이 아니라 "현재 코드"다.
- 문서가 코드보다 뒤처지면, 코드 확인 후 문서를 바로 갱신한다.
- 기능 단위 완료는 백엔드 완료나 iOS 완료가 아니라 "양쪽 연동 확인 완료"를 기준으로 본다.

| 목표 | CRUD, progress 구현 | 생성/목록/진행/포기 연결 | 연동 완료 | 상세/수정 활용 확대 가능 |
| 회고/변화 분석 | weekly-summary, change-analysis API 구현 완료 | WeeklyRetrospectiveView, ChangeAnalysisView 실데이터 연동 | 연동 완료 | HistoryCalendarView 데이터 연결 남음 |
| 알림/인사이트 | FCM/인사이트 미완 | Firebase 초기 연결 수준 | 미완 | 공통 이벤트 설계 필요 |

## 작업 단위 정의

앞으로는 기능을 아래 5단계로 나눠서 움직인다.

### 1. 계약 확정

- 백엔드: 엔드포인트, 필수 필드, 에러 코드, 정렬/필터 규칙 확정
- iOS: 필요한 화면 상태, 빈 상태, 로딩 상태, 에러 상태 확정
- 문서: 필요 시 `API_DESIGN.md` 또는 실행 문서에 변경사항 반영

완료 기준:
- iOS가 임시 필드명이나 추측으로 구현하지 않아도 된다.

### 2. 백엔드 구현

- Controller, DTO, Service, Repository 구현
- 핵심 단위 테스트 또는 컨트롤러 테스트 추가
- 인증/권한 경계가 있으면 보안 테스트 포함

완료 기준:
- Postman/MockMvc/테스트 기준으로 계약이 실제 응답으로 확인된다.

### 3. iOS 연동

- `APIEndpoint`, 모델, ViewModel, 화면 상태 연결
- 성공/실패/빈 상태 처리
- 백엔드 응답 형식과 동일한 필드 기준으로 디코딩 확인

완료 기준:
- 하드코딩 없이 실데이터로 흐름이 재현된다.

### 4. 연동 검증

- 같은 시나리오를 백엔드와 iOS에서 하나의 사용자 흐름으로 점검
- 생성 → 조회 반영, 삭제 → 목록 반영, 인증 실패 → 사용자 메시지 확인

완료 기준:
- 사용자 입장에서 "기능이 된다"가 확인된다.

### 5. 문서/회귀 반영

- `CURRENT_STATUS.md` 반영
- 관련 TODO 문서 갱신
- 신규 계약이면 이 문서의 연동 상태 표도 업데이트

완료 기준:
- 다음 작업자가 현재 기준선을 문서만 보고도 이해할 수 있다.

## 기능 개발 순서 규칙

기능 하나를 추가할 때는 아래 순서를 기본값으로 사용한다.

1. 백엔드 계약/DTO 확정
2. 백엔드 테스트 작성 및 API 구현
3. iOS `APIEndpoint` 및 모델 반영
4. iOS ViewModel 연결
5. iOS 화면/상태 처리
6. 양쪽 문서 갱신

예외:
- 이미 백엔드 계약이 안정적인 경우 iOS 작업을 먼저 해도 된다.
- 단, iOS가 임시 mock 데이터에 의존해 먼저 진행한 경우 "연동 전환 체크"를 반드시 남긴다.

## 공통 완료 체크리스트

기능 하나가 끝났다고 말하려면 아래를 같이 만족해야 한다.

- 백엔드 엔드포인트가 실제 테스트로 검증됨
- iOS에서 실데이터로 호출됨
- 인증 실패 시 사용자 흐름이 깨지지 않음
- 빈 데이터 상태가 정의됨
- 삭제/재시도/실패 중 해당 기능에 필요한 상태가 정리됨
- 관련 문서가 업데이트됨

## 지금 확인된 갭

### 1. 신체 측정 그래프용 API는 백엔드에 있고 iOS 활용은 아직 약함

- 백엔드에는 `GET /api/v1/body-measurements/range`
- 백엔드에는 `GET /api/v1/body-measurements/at-or-before`
- iOS는 현재 목록/추가/삭제 중심으로 사용 중

의미:
- 다음 iOS 그래프 작업은 새 API 설계 없이 바로 붙일 가능성이 높다.

### 2. 목표 상세/수정 API는 백엔드에 있고 iOS는 생성/목록/진행 중심

- 백엔드에는 `GET /api/v1/goals/{id}`, `PATCH /api/v1/goals/{id}`
- iOS는 현재 `getGoals`, `createGoal`, `getGoalProgress`, `deleteGoal` 중심

의미:
- 목표 수정 UX를 추가할 때 서버 작업보다 iOS 작업 비중이 더 클 수 있다.

### 3. 진행 사진은 연동돼 있지만 사용성 후속 작업이 남음

- 백엔드: 업로드 URL 발급, 메타 등록, 목록 조회
- iOS: 업로드/목록/상세 연결
- 미완: 삭제, 비교, 썸네일 대응, 실패 복구 UX

의미:
- 다음 작업은 계약 신규 정의보다 UX와 후처리 연동 보강이다.

## 다음 공통 작업 순서

### 1. 진행 사진 후속 작업을 하나의 슬라이스로 묶기

- 백엔드: 썸네일/업로드 완료 검증/필요 시 삭제 API
- iOS: 삭제/비교/업로드 실패 복구 UX
- 문서: 진행 사진 플로우를 양 플랫폼 기준으로 재정리

### 3. 권한 경계 테스트와 iOS 에러 처리 기준 맞추기

- 백엔드: 타 사용자 데이터 접근 시나리오 테스트
- iOS: `401`, `403`, `404` 대응 메시지와 재로그인 흐름 정리

## 변경 규칙

- 백엔드가 응답 필드명, 경로, 상태 코드를 바꾸면 같은 작업에서 iOS와 문서를 함께 수정한다.
- iOS가 신규 화면 상태를 요구하면 백엔드 응답이 그 상태를 지원하는지 먼저 확인한다.
- TODO 문서에는 "한 플랫폼 단독 작업"보다 "연동 단위 작업"을 우선으로 쓴다.
